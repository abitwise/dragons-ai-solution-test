      { score: final state.score, turns: final state.turn, reason: END.GAME_OVER }.
    - Shop-phase drain: getShop returns a buyable item, chooseShopPurchase picks it, api.buy is
      called, applyBuyResult folds the raw BuyResult (score/highScore preserved from prior state),
      the shop is re-fetched, and the drain stops when chooseShopPurchase returns null.
    - Shop-phase drain stops on a buy reporting shoppingSuccess:false (no infinite re-buy when gold
      is insufficient but the strategy keeps recommending the same item).
    - Fresh ads before solve: assert via fake.calls ordering that getMessages is called after the
      shop phase (getShop/buy precede the getMessages preceding each solve) — proving expiresIn is
      current at decision time (LOOP-03).
    - chooseAd returns null (truly empty board) → the iteration performs no turn-consuming solve and
      does not crash; the run continues (in this plan, drive such a script to a later lives:0 solve
      so it still terminates via GAME_OVER — the no-progress termination of a permanently-empty
      board is plan 03-02's concern).
    - Score preservation: a script with a buy (raw BuyResult, no score/highScore) followed by a
      solve proves the final GameReport.score came from the threaded state, NOT zeroed by the buy.
  </behavior>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| runner → ApiClient → live game API | The runner trusts typed models the ApiClient already validated/coerced (zod at the api.ts boundary, Phase 1). The runner itself crosses NO new trust boundary: it never touches `fetch`, parses raw JSON, or reads env. |
| API response `turn`/`lives`/`gold` fields → loop control | The loop's `while (state.lives > 0)` and (in 03-02) the turn cap consume API-provided numbers. A hostile/buggy API returning a never-decreasing `lives` could loop — bounded by 03-02's dual guards (out of scope here). |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Denial of Service | playGame `while (state.lives > 0)` loop | accept (here) / mitigate (03-02) | This plan ends only on `lives === 0`. The unbounded-loop risk (a flat/never-decreasing `lives`/`turn`) is the explicit subject of plan 03-02's MAX_TURN cap (D-05) + no-progress guard (D-06/D-07). The `END.TURN_CAP`/`NO_PROGRESS` constants are declared here so 03-02 only adds the checks. |
| T-03-02 | Denial of Service | shop-phase inner drain loop | mitigate | The drain stops on `chooseShopPurchase` returning `null` OR a buy with `shoppingSuccess:false` (D-02) — so an unaffordable-but-recommended item cannot trigger an infinite re-buy. Verified by the "stops the shop drain when a buy reports shoppingSuccess:false" test. |
| T-03-03 | Tampering | API `score`/`level` fields via merge | mitigate | State advances ONLY through `applySolveResult`/`applyBuyResult` (D-04), which preserve `score`/`highScore` across a buy and `level` across a solve — a raw `BuyResult` (no score) can never silently zero the reported score. | T-03-SC | Tampering | npm/pip/cargo installs | n/a | This plan installs NO new packages — `runner.ts`/`runner.test.ts` compose existing in-repo modules + already-installed Vitest. No package-legitimacy checkpoint required. |
