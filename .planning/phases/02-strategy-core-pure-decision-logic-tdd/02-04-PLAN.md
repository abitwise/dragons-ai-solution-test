<objective>
Drive the two state-merge helpers test-first (STRAT-06): `applySolveResult(state, SolveResult): GameState` carries `level` forward (a solve result omits `level`), and `applyBuyResult(state, BuyResult): GameState` carries `score`/`highScore` forward (a buy result omits both). Both are pure â€” they return a new `GameState` and never mutate the prior one.

Purpose: The solve/buy responses are intentionally asymmetric (types.ts documents it; FEATURES.md lines 56-66 verify it live). A naive "replace state with the response" loses a field every turn â€” `level` would reset after each solve, and `score`/`highScore` would reset after each buy. The buy case is especially load-bearing: `api.ts buy()` (lines 209-225) returns a `GameState` with `score: 0, highScore: 0` placeholders because the client has no prior state to merge â€” so the merge helper here is what restores them. Getting this wrong silently corrupts the final score the whole project reports.

Output: `applySolveResult` and `applyBuyResult` in `src/strategy.ts`; tests in `src/strategy.test.ts` proving each preserves the field its response omits, and proving non-mutation of the prior state.
</objective>
