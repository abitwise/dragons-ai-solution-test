<objective>
Close the three open Phase-2 items from verification + code review without re-planning the already-built, passing 02-01..02-04 work. Two are correctness hardening in the pure decision core; one is the inter-phase seam contract fix the verifier flagged as `human_needed` (WR-02).

1. **WR-01 (non-finite `reward`):** `Ad.reward` is coerced from a free-text wire string in `api.ts`; a failed coercion yields `NaN`. A `NaN` EV makes both `evDiff > 0` and `evDiff < 0` false in `preferAd`, so selection silently falls through to the tiebreak and can pick a garbage-EV ad over a legitimately high-EV one. Fix: drop non-finite-reward ads in `filterEligibleAds` AND in the `chooseAd` fallback `solvable` filter — keeping the primary and fallback filters in lock-step — so a non-finite-reward ad never reaches selection.

2. **WR-03 (non-finite `cost`):** `ShopItem.cost` flows from the same boundary. A `NaN` `hpot` cost makes `hpot.cost <= state.gold` false, so a needed heal silently does not fire (the bot quietly stops healing). Fix: guard cost reads with `Number.isFinite(item.cost)` in BOTH the heal lookup and the upgrade filter, treating a non-finite cost as unaffordable (degrade to null / exclude, never throw).

3. **WR-02 (seam contract):** `solve()` returns a raw `SolveResult` (runner merges via `applySolveResult`), but `buy()` returns a PRE-merged `GameState` with `score: 0`/`highScore: 0` placeholders — making `applyBuyResult` (the documented score-protector) unreachable through the injected `ApiClient` seam. A Phase-3 runner would either adopt the zeroed score or hit a type error. Fix (locked decision): make `buy()` symmetric with `solve()` — `ApiClient.buy()` returns `Promise<BuyResult>`, `HttpApiClient.buy()` returns the raw validated `BuyResult` (deleting the GameState construction + placeholders), and the runner folds it via `applyBuyResult`. `applyBuyResult`'s logic is already correct and needs NO change — only its JSDoc and the seam types/tests change.

Purpose: These touch the load-bearing decision loop CLAUDE.md says "must work" and the seam that protects the final reported score. Getting WR-02 wrong means the score-protect logic is dead code and Phase 3 silently zeroes the score.

Output: hardened `strategy.ts` (pure, never throws, degrades to skip/null on non-finite inputs); a symmetric `buy()` seam across `types.ts`/`api.ts`/`fake-api-client.ts` returning the raw `BuyResult`; updated tests proving the new contract and the regression guards; updated JSDoc removing the now-false "buy returns a merged GameState / score:0 placeholder" claims.
</objective>
