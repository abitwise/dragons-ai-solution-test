<objective>
Implement `decodeAd` — the pure, cross-field encryption-decode step — fully test-first (RED → GREEN → REFACTOR). This is the single most test-worthy unit in the phase and the highest-ROI differentiator (it unlocks high-reward encrypted ads). Per D-03, decoding is a SEPARATE step from zod validation (NOT a `.transform`), because one `encrypted` flag governs three fields (`adId`, `message`, `probability`) — a cross-field operation that is awkward in per-field zod transforms and is cleanest to unit-test in isolation.

The contract (D-08/D-09): for a KNOWN scheme (`encrypted:1` = Base64, `encrypted:2` = ROT13), decode all three string fields TOGETHER and CLEAR the `encrypted` flag. For an UNKNOWN scheme (e.g. `3`) or a decode failure (corrupt Base64), pass the ad through UNCHANGED with its `encrypted` flag still set — never drop, never throw, never partially decode (PITFALLS #1: "decode all three fields or none"). Strategy's filter (Phase 2) then drops still-flagged ads and the loop logs the skip.

Purpose: A correct, isolated, exhaustively-tested decode function that api.ts will call after zod validates wire shape. Implements API-05.
Output: `src/decode.ts` (`decodeAd` + base64/rot13 helpers) and `src/decode.test.ts`, all green, no network.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02 | Tampering / Injection | decodeAd handling of untrusted ad strings | mitigate | Decoded text is treated as pure data — never `eval`'d, never interpolated into a shell/template/URL inside decode.ts; Base64 is regex-validated before decode; a corrupt/garbage payload returns the ad unchanged rather than producing a mis-decoded id |
| T-01-03 | Denial of Service | malformed/corrupt encrypted payload | mitigate | decodeAd never throws on bad input (all-or-none pass-through), so one corrupt ad cannot crash the process; pure synchronous function with no unbounded work |
