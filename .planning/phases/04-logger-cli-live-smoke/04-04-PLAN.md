<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Live API response → bot | The only live network call in the project; untrusted JSON (ad messages, encrypted text, shop names) crosses into the bot at `api.ts` |
| (only outbound network) | A single outbound HTTPS conversation to the game API; nothing inbound, no auth, no DB |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-06 | Spoofing/Tampering | The live API host / response integrity | accept | The base URL is the fixed non-www `https://dragonsofmugloar.com/api/v2` over HTTPS, read ONCE at construction in `api.ts` (never from a response — SSRF guard, locked carry-forward). This is a public game API with no auth and no secrets; a hostile response can at worst end the run (BoundaryError → exit 2). Acceptable for a local demo smoke. |
| T-04-01 | Tampering | Untrusted live ad/shop strings rendered in narration | mitigate | Already mitigated upstream (Plans 01/02): untrusted strings ride as structured pino fields, not concatenated into messages, so live response data cannot forge log lines. This task observes that mitigation holding against REAL data. |
| T-04-02 | Information Disclosure | DEBUG dump of raw live API objects | accept | The game API carries no credentials/PII (stated assumption); raw objects appear only at DEBUG. Running `LOG_LEVEL=debug` is the operator's deliberate choice on a local terminal. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No new dependencies installed this phase. No install task in this plan. |
</threat_model>
